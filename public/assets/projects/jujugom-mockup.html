하나의 답이 아니라 여러 관점이 필요했습니다

03
폴백 체인 설계

"안 될 때"를 먼저 설계했습니다

AI 서비스의 진짜 기획은 "잘 될 때"가 아니라 "안 될 때"입니다. Finnhub API 키가 없거나, BeatTheCap 서버가 느리거나, 한국 종목이라 데이터가 없을 때 — 사용자는 "왜 안 되지?"가 아니라 "그냥 되네"여야 합니다.

!

문제
Finnhub 무료 티어 한계 (60 req/min), 한국 종목 미지원, BeatTheCap 첫 로딩 60초

→

설계 원칙
"모든 데이터 경로에 최소 2단계 폴백. 빈 화면은 절대 보여주지 않는다."

→

결과
API 키 없이도 yfinance로 시세·차트·재무 제공. 한국 종목 자동 .KS 변환

3-Tier Fallback Chain

Tier 1 · Finnhub + BeatTheCap

API 키 있을 때 우선 사용. 캐시 7일

Tier 2 · yfinance

Tier 1 실패 시 자동 전환. 한국 종목(.KS) 지원

Tier 3 · Graceful Empty

빈 데이터지만 에러 없이. "데이터 없음" 상태 표시

기술 선택

왜 이 조합인가

Claude CLI (subprocess)

Anthropic SDK 대신 CLI subprocess를 선택했습니다.

+API 키 관리 불필요 (CLI 인증 활용)
+스트리밍 출력 → SSE로 직접 파이프
−프로세스 오버헤드, 에러 핸들링 복잡

SSE (Server-Sent Events)

WebSocket 대신 SSE를 선택했습니다.

+단방향 스트리밍에 최적 (분석 결과 전달)
+HTTP/2 호환, 재연결 자동
−양방향 통신 불가 (필요 없었음)

yfinance 폴백

유료 API (Finnhub) 의존도를 줄이기 위해 무료 폴백을 설계했습니다.

+API 키 없이도 서비스 가동
+한국 종목 지원 (KRX .KS/.KQ 자동 변환)
−속도 느림 (sync → thread pool)

React 19 + Vite + shadcn/ui

AED와 동일한 선택. 1인 개발에서 속도가 핵심이었습니다.

+HMR 즉시 반영, 빌드 빠름
+shadcn/ui로 디자이너 없이 일관된 UI
+PWA 지원 (모바일 홈 화면 추가)

Key Learnings

이 프로젝트에서 배운 것

1. AI에게 일을 시키는 건
팀 빌딩이다

"분석해줘"라는 한 마디가 6개 역할 분담 + 결과 합산 로직 + 가중치 설계로 분해됐습니다. 사람 팀에서 "각자 알아서 해"가 안 되듯, AI에게도 명확한 역할 정의와 결과 포맷이 필요합니다.

2. 불확실한 출력을
확실하게 보여주는 법

AI가 매번 다른 답을 내는 건 버그가 아니라 특성입니다. "정답"을 주는 대신 "4명이 이렇게 생각한다"는 구조가 사용자 신뢰를 높였습니다. 불확실성을 숨기지 않고 다양성으로 전환한 설계입니다.

3. 폴백은 보험이 아니라
기본 아키텍처다

"나중에 폴백 넣자"가 아니라 처음부터 모든 데이터 경로에 2단계 이상의 대안을 설계했습니다. 특히 외부 API에 의존하는 서비스에서, 폴백 없는 기능은 미완성입니다.
