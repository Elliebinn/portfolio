Background

전략은 쏟아지는데,
보여줄 화면이 없었습니다

퀀트 리서치팀의 전략 생성 엔진이 가동되자
전략 수는 10개에서 수백 개 단위로 빠르게 늘어났습니다.
늘어난 건 전략 수만이 아니었습니다.

전략마다 샤프지수, CAGR, MDD, Calmar 등 10개 이상의 성과 지표를 계산해야 했고,
종목별 매매 사이클 분석, 이격도 차트, 벤치마크 대비 IS/OS 기간 분리 검증까지.
하나의 전략을 제대로 평가하기 위한 분석 깊이 자체가 달라지고 있었습니다.

기존 NiceGUI 화면은 "전략이 올라갔는지 확인하는" 수준이었고,
리서처마다 자기 환경에서 필요한 것만 따로 확인하는 정도였습니다.
대표님이 외부 투자자 미팅에서 전략을 보여주고 싶어도,
보여줄 수 있는 화면 자체가 없었습니다.

My Role

운용 경험을 가진 기획자가 목업을 먼저 만들어 제안했습니다

AI 공모펀드 운용 경험에서 얻은 감각으로 HTML 목업을 먼저 만들어 공유했습니다. 대표님의 "이 정도만 만들어져도 좋겠다"는 한마디가 프로젝트의 시작이 됐습니다. PRD 작성부터 화면 설계, 프론트엔드 구현까지 직접 수행했습니다.

Problem → Solution

4가지 문제, 하나의 플랫폼으로

01
기준 불일치

같은 전략을 보고도 판단이 달랐습니다

리서처마다 샤프지수 기준이 달랐고, "이 전략이 좋다"의 근거가 공유되지 않는 상황.

→ 지표 산식을 표준화하고 glossary로 정의를 통일. 한 대시보드에서 동일 기준으로 비교.

02
화면 부재

전략의 가치를 보여줄 수 없었습니다

차트가 세로로 나열된 기존 화면. 외부 미팅에서 전략의 강점을 설명할 근거 화면이 없었습니다.

→ 금융 대시보드 14개를 조사해 정보 위계를 재설계. 핵심 지표 → 상세 분석으로 이어지는 구조.

03
실행 불가

아이디어가 화면이 될 수 없었습니다

디자이너, 프론트엔드 개발자 없는 팀. 기획서가 문서로 끝나는 상황이었습니다.

→ HTML 목업 32개를 직접 만들고 코드로 전환하는 파이프라인 구축. 목업이 곧 제품.

04
검증 불가

매매 논리를 검증할 수 없었습니다

"왜 이 종목을 이 타이밍에 샀는지" 확인할 방법 없이 성과 숫자만 보는 상황.

→ 종목별 드릴다운으로 매매 사이클마다 이격도·캔들스틱 분석. 알고리즘 판단 근거를 시각적으로 검증.

전략을 만들고 성과를 검증하는 사람. 매일 사용.

"이 전략이 왜 3월에 손실을 냈는지 알고 싶어요.
어떤 종목 때문인지, 그 종목을 왜 그 타이밍에 샀는지까지 추적할 수 있어야 합니다."

펀드매니저

전략을 실전 포트폴리오에 투입할지 판단하는 사람. 주간 리뷰 시 사용.

"이 알고리즘이 지금도 제대로 작동하고 있는 건지 빠르게 확인하고 싶어요.
성과가 괜찮으면 실전에 넣을 건데, 판단할 수 있는 화면이 필요합니다."

User Flow

리서처의 분석 플로우

01

전체 건강도 진단

전체 전략의 성과 지표가 한눈에 보입니다.
품질이 떨어지는 전략이 늘고 있는지, 위험 대비 수익이 비정상인 전략이 있는지를 대시보드에서 즉시 파악합니다.

02

이상치 추적

문제 있는 전략을 클릭하면 성과요약으로 진입합니다.
학습 기간에는 성과가 좋았지만 검증 기간에 급락한 전략인지를 분리해서 확인할 수 있습니다.

03

원인 종목 식별

기여수익률 테이블에서 어떤 종목이 전체 손실에 가장 크게 기여했는지를 확인합니다.
해당 종목을 클릭하면 전체 거래 내역이 열립니다.

04

매매 논리 검증

손실 사이클의 매매 차트를 열어 알고리즘이 어떤 기준으로 매수했는지를 확인합니다.
의도한 기준과 실제 진입 시점이 달랐다면 그 원인까지 추적할 수 있습니다.

02
운용 관점의 날짜 탐색

달력을 바꿔달라는 요청 뒤에 있던 것

운용을 해본 사람은 압니다.
전략을 평가할 때 "성과가 좋다/나쁘다"만으로는 부족합니다. 특정 날짜에 어떤 종목을 얼마나 들고 있었는지, 리밸런싱이 일어난 날에 비중이 어떻게 바뀌었는지를 직접 확인해야 판단이 가능합니다.

1

비중 차트 클릭 = Holdings 동기화

포트폴리오 비중 차트의 아무 지점을 클릭하면 그 날짜의 보유 종목 테이블로 즉시 이동. 차트와 테이블이 같은 날짜를 바라봅니다.

2

리밸런싱 일자 네비게이션

날짜를 하루씩 넘기는 것만으로는 부족합니다. 운용역은 "다음 리밸런싱 날"로 바로 점프하고 싶어합니다. 리밸런싱 전용 버튼(◀R / R▶)을 추가해서 비중 변동이 실제로 일어난 날만 빠르게 탐색할 수 있습니다.

3

성과에서 매매탐색으로의 크로스탭

성과 차트에서 의문이 생긴 날짜를 클릭하면 그 시점의 보유 종목과 누적 기여를 보여주는 스냅샷 바가 나타납니다. "매매탐색에서 자세히 →" 클릭 한 번으로 해당 날짜의 종목별 드릴다운으로 진입합니다.

지표 품질 일원화

UI를 다듬는 것보다 먼저 한 일.
전체 지표의 계산 공식을 검증. 샤프지수가 무위험 수익률(Rf)을 차감하지 않고 있었고 CAGR은 역일(365.25)과 영업일(252)이 혼용되고 있었습니다. 전수 감사 후 10개 지표 공식을 PRD에 표준 정의하고 461개 지표를 단일 glossary로 통합했습니다.

Tooltip 구조

샤프지수

위험 대비 초과수익을 측정하는 지표. 1 이상이면 양호.

formula

sqrt(252) x mean(r - rf) / std(r)

rf = 2% 연간 (0.02/252 일일)

04
프레임워크 전환

도구가 한계에 부딪히면 바꿉니다

NiceGUI(Python UI)는 빠르게 만들 수 있지만 차트 인터랙션에 한계가 있었습니다. Next.js로 전환했으나 내부 도구에 SSR은 불필요하고 번들이 과잉. Vite + React로 재전환했습니다.
HMR 속도가 곧 개발 속도입니다. "틀린 선택을 빠르게 인정하고, 더 맞는 도구로 옮기는 것"이 1인 개발에서 가능한 판단.

전환 과정

1

NiceGUI (Python)

빠른 프로토타이핑, 차트 인터랙션 한계

2

Next.js

SSR 불필요, 번들 과잉

3

Vite + React CSA

HMR 즉시, 내부 도구에 최적

05
상세한 매매 검증

종목 하나를 클릭하면 그 전략의 논리가 보입니다

기여수익률 테이블에서 종목을 클릭하면 해당 종목의 전체 거래 내역이 펼쳐집니다. 주가 차트 위에 매수/매도 마커가 찍히고, 수익 구간은 녹색, 손실 구간은 붉은색으로 하이라이트됩니다. 사이클을 클릭하면 그 구간의 캔들스틱 + 이격도 차트 + 거래 상세가 우측에 슬라이드로 열립니다.

1

종목 전체 거래 내역

주가(좌축) + 포트폴리오 NAV(우축) + 벤치마크를 한 차트에. 매수/매도 마커로 진입/청산 시점을 즉시 파악.

2

사이클별 성과 분석

매매 사이클(매수에서 매도) 단위로 수익률과 영업일, 고점대비 낙폭을 비교합니다. 클릭하면 해당 구간의 캔들스틱으로 확대.

3

이격도 분석

주가와 이동평균의 이격도를 원본/변동성 조정 두 가지로 비교합니다. 몇 단계에서 진입했고 밴드가 얼마나 축소되었는지를 시각화.

4

거래 상세

신규매수/추가매수/부분매도/전체매도.
각 거래의 가격과 수량, 보유 수량, 진입 단계와 사유까지 한 테이블에.

Phase 1 관찰

리서처 옆에 앉다

리서처 옆에서 실제 분석 워크플로우를 관찰. 전략 확인부터 성과 평가까지 어디서 멈추는지를 파악하고 필요한 화면의 우선순위를 도출.

Phase 2 구조 설계

분류 체계부터 세우다

NiceGUI 위에서 태그 시스템과 유니버스 필터, 리더보드, 동적 벤치마크 등 핵심 구조를 먼저 구현.

Phase 3 화면 설계

32개 목업으로 방향을 잡다

디자이너 없이 32개+ HTML 목업을 직접 제작. A~D 시안 비교 후 리서처에게 바로 공유하며 피드백.

Phase 4 피드백

디자이너 평가 2.5점을 받다

같은 회사 UX/UI 디자이너 2명에게 종합 평가 요청. 7개 REQ로 분해하고 14개 레퍼런스 조사 후 전면 리디자인.

Phase 5 마이그레이션

예정보다 빠르게 전환하다

사용자 증가와 분석 기능 복잡화로 서버 부하 한계에 도달. React+Vite로 전환. PRD와 디자인 토큰을 재활용해 5일 만에 이전.

HTML 목업 32개를 직접 만들고 바로 코드로 전환했습니다.
기획 의도가 그대로 화면에 반영되니
결과물이 달라서 수정하는 과정이 사라졌습니다.
프론트엔드 개발자에게 설명하고 기다리고
다시 확인하는 사이클이 필요 없었습니다.
목업에서 보이는 것이 그대로 제품이 됩니다.
"이거 이 뜻이 아닌데요" 회의가 구조적으로 없어졌습니다.

"달력 바꿔주세요" 뒤의 진짜 질문

리서처가 "날짜 선택이 불편하다"고 말했을 때
보이는 문제는 달력 위젯이었습니다.
하지만 직접 펀드를 운용해본 경험에서
진짜 필요한 건 "다음 리밸런싱 날짜로 점프"라는 걸 알 수 있었습니다.
포트폴리오 비중 차트를 클릭하면
그 날짜의 보유 종목으로 동기화되는 기능도 같은 맥락입니다.
요청에 없었지만 운용자라면 당연히 필요한 것입니다.

샤프지수에서 Rf가 빠져 있었다

성과요약 화면에서 숫자를 보다가
"이 수치가 뭔가 이상하다"는 감각이 들었습니다.
확인해보니 지표 계산에 오류가 있었습니다.
화면에서 직접 숫자를 보지 않았다면
발견할 수 없었던 문제였습니다.
전수 감사를 돌려 10개 지표 공식을 표준화하고 일괄 수정했습니다.
기획서만 쓰는 PM은 이 숫자를 볼 기회 자체가 없습니다.
