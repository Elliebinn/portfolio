## 프론트엔드에는 테스트 코드가 없었어요

백엔드는 테스트를 돌리면 돼요. 코드를 바꾸고, 테스트를 실행하고, 통과하면 괜찮은 거죠. 근데 프론트엔드는요?

우리 분석 도구는 [NiceGUI](https://nicegui.io/)라는 Python 기반 웹 프레임워크로 만들어져 있어요. UI 쪽에 유닛 테스트를 넣을 수 있는 구조가 아니었어요. 그래서 뭔가 바꾸면 확인하는 방법이 하나뿐이었죠 — **브라우저에서 직접 눈으로 보는 것.**

차트 색상 바꾸고 → 브라우저 열어서 확인. 테이블 정렬 고치고 → 새로고침해서 확인. 사이드바 수정하고 → 페이지 이동해가며 확인.

이것 자체는 괜찮아요. 문제는 그 다음이었어요.

---

## 진짜 귀찮은 건 "공유"였어요

저는 혼자 일하는 게 아니잖아요. 뭔가를 바꾸면 팀에 공유해야 해요.

클릭업에 업데이트를 올리거나, 다른 팀원에게 "이거 이렇게 바뀌었어요"라고 알려줘야 하는데 — 이게 매번 너무 번거로운 거예요.

일단 **변경 전 모습을 기억 못 해요.** 코드를 수정하고 나면 이전 화면이 어떻게 생겼었는지 정확히 떠올릴 수가 없어요. "아 그전에는 여백이 좀 더 넓었나? 글자 크기가 달랐나?" 이런 게 모호해지거든요.

그래서 공유할 때마다 이런 과정을 반복하게 돼요.

1. 바꾸기 전에 스크린샷 찍기 (기억나면)
2. 코드 수정하기
3. 바꾼 후에 스크린샷 찍기
4. 뭐가 바뀌었는지 텍스트로 정리하기
5. 문제가 뭐였고, 어떻게 해결했는지 적기
6. 스크린샷이랑 같이 클릭업에 올리기

작은 거 하나 수정하는 데 실제 코드 수정은 5분이면 되는데, **공유용으로 정리하는 데 20분이 걸리는** 상황. 그리고 솔직히 바빠지면 1번을 빼먹어요. 바꾸기 전에 스크린샷을 안 찍는 거죠. 그러면 before/after 비교 자체가 불가능해져요.

---

## 수동 캡처의 한계

처음에는 그냥 참고 했어요. 매번 수동으로.

근데 변경이 쌓이면서 점점 힘들어졌어요. 20개 넘는 페이지를 리디자인하면서, 한 번에 여러 페이지가 영향 받는 경우가 많았거든요. 예를 들어 사이드바 스타일을 바꾸면 모든 페이지의 사이드바가 바뀌잖아요. 그러면 대표로 몇 개만 캡처해야 하는데 — 어떤 페이지를 찍어야 하는지도 매번 고민이에요.

그리고 인터랙션이 있는 변경은 더 골치 아팠어요. "차트를 클릭하면 오버레이가 뜨는 기능"을 추가했다고 해볼게요. 기본 상태 스크린샷만으로는 이 변경이 안 보여요. 실제로 클릭해서 오버레이가 뜬 상태를 캡처해야 하거든요. 이런 시나리오를 매번 수동으로 재현하는 건 정말 귀찮았어요.

---

## "이거 자동화해야겠다"

그래서 생각한 게, **코드 변경(git diff)을 기반으로 변경 전후 스크린샷을 자동으로 찍고, 리포트까지 생성하면 안 될까?** 였어요.

원리는 간단해요.

1. 내가 코드를 수정한 상태에서 `/frontend-checkpoint` 을 실행하면
2. `git stash`로 수정 전 상태를 복원하고 → 스크린샷 촬영
3. `git stash pop`으로 수정 후 상태를 적용하고 → 같은 화면 다시 촬영
4. before/after를 비교해서 리포트를 자동 생성

이렇게 하면 바꾸기 전 스크린샷을 깜빡 안 찍어도 괜찮아요. git이 이전 상태를 갖고 있으니까, 언제든 복원해서 찍을 수 있거든요.

---

## 만들면서 부딪힌 것들

원리는 간단한데, 실제로 만들려니까 생각보다 복잡한 부분이 많았어요.

### "어떤 페이지를 찍어야 하지?"

코드를 수정하면 `git diff`에 변경된 파일이 나와요. 근데 `frontend/layout.py`를 수정했다고 해서 layout 페이지를 찍는 게 아니잖아요. 이 파일은 모든 페이지에 영향을 주거든요.

그래서 **변경 파일 → 영향받는 페이지 라우트**를 매핑하는 로직이 필요했어요. `@ui.page` 데코레이터를 추적하고, import 관계를 따라가서 어떤 페이지가 영향을 받는지 자동으로 파악하게 만들었어요.

### "기본 화면만 찍으면 변경이 안 보이는데?"

이게 제일 고민이었어요. 아까 말한 것처럼, 인터랙션이 있는 변경은 기본 상태 캡처로는 안 보여요.

그래서 **동적 시나리오 탐색**이라는 걸 넣었어요. [Playwright](https://playwright.dev/)라는 브라우저 자동화 도구로 페이지를 열고, DOM 구조를 분석해서 클릭할 수 있는 요소(탭, 버튼, 드롭다운, 테이블 행 등)를 자동으로 찾아요. 그걸 하나씩 클릭하면서 상태가 변하면 스크린샷을 찍는 거죠.

더 중요한 건, `git diff`를 분석해서 **새로 추가된 기능**을 찾아내는 거였어요. `on_click`이 추가됐으면 그 요소를 클릭하는 시나리오를, 새 차트가 추가됐으면 그 차트 영역을 캡처하는 시나리오를 자동으로 만들어요. 새 기능이 추가됐는데 after 스크린샷에 안 찍히면 의미가 없으니까요.

### "스크린샷이 너무 많아"

시나리오를 자동으로 만들다 보면 before/after 쌍이 엄청 많아져요. 근데 그중 90%는 변화가 없거나, 다른 시나리오랑 같은 변화예요.

그래서 **필터링** 단계를 넣었어요. before/after를 비교해서 의미 있는 차이가 있는 것만 남기고, 비슷한 패턴은 대표 하나만 남기고 나머지는 제거해요. 리포트에는 "이런 시나리오들은 동일 패턴이라 대표 3개만 수록했다"고 표시하고요.

## 결과: 자동 생성되는 리포트

이제 `/frontend-checkpoint "사이드바 active 하이라이트 수정"` 한 줄이면 이런 게 자동으로 만들어져요.

```
.screenshots/checkpoints/
└── 2026-03-23_web-redesign_title-sidebar-unify/
    ├── REPORT.md          ← 변경 요약 + before/after 비교
    ├── meta.json          ← 변경 파일, 브랜치, 커밋 등 메타데이터
    ├── overview_before.png
    ├── overview_after.png
    ├── result-list_before.png
    ├── result-list_after.png
    └── ...
```

REPORT.md에는 이런 내용이 들어가요.

- **변경 요약**: 뭘 바꿨는지 한 줄로
- **변경사항 상세**: 파일별로 뭐가 달라졌는지 (코드 diff 기반)
- **Before/After 비교**: 스크린샷 + "여기가 달라졌다" 설명
- **제외된 시나리오**: 같은 패턴이라 뺀 것들 목록

이걸 그대로 클릭업에 올리면 끝이에요. 따로 정리할 필요가 없어요.

실제로 사이드바 하이라이트를 고쳤을 때 자동 생성된 리포트를 보면, "10개 파일을 수정했고, 7개 페이지에 영향이 있었는데, 대표 3개 페이지의 before/after와 상세 변경사항"이 깔끔하게 정리되어 나왔어요. 수동으로 했으면 30분은 걸렸을 작업이요.

## 이 스킬을 만들면서 느낀 것

### 반복 작업이 보이면, 도구를 만들어야 해요

스크린샷 찍고 정리하는 건 한 번은 괜찮아요. 근데 매번 하는 거면 그건 자동화 대상이에요. 귀찮은 걸 참는 게 아니라, "이 귀찮음이 반복되나?"를 먼저 판단하는 게 중요하더라고요.

### 공유 비용을 줄이면 팀 전체가 빨라져요

코드 수정 자체보다 "이걸 어떻게 설명하지?"에 시간을 더 쓰고 있었어요. 리포트가 자동으로 나오니까 공유 빈도가 올라갔고, 팀원들도 "무슨 변경이 있었는지" 더 빨리 파악할 수 있게 됐어요. 작은 자동화가 팀 소통 비용을 확 줄여주더라고요.

### AI 스킬이라는 게 결국 "워크플로우 설계"예요

이 스킬은 AI가 코드를 대신 짜주는 게 아니에요. **"git stash → 촬영 → stash pop → 촬영 → 비교 → 리포트"라는 워크플로우를 설계**한 거예요. AI는 그 워크플로우를 실행하는 역할이고요. 잘 만든 워크플로우가 있으면 AI가 더 정확하게 일하고, 워크플로우가 모호하면 AI도 엉뚱한 결과를 내요.

이건 서비스 기획이랑 똑같은 원리예요. 사용자 플로우를 잘 설계하면 서비스가 자연스럽게 돌아가는 것처럼, AI 도구도 플로우를 잘 짜면 원하는 결과가 나와요.

---

## 기술적으로 풀어야 했던 문제들

나중에 비슷한 걸 만들고 싶은 분들을 위해 정리하면:

| 문제 | 해결 |
|------|------|
| 변경 파일 → 영향 페이지 매핑 | `@ui.page` 데코레이터 + import 관계 추적 |
| 변경 전 상태 복원 | `git stash` / `git stash pop` 활용 |
| 인터랙션 캡처 | Playwright DOM 스냅샷 분석 → 클릭 가능 요소 자동 탐색 |
| 새 기능 누락 방지 | `git diff`에서 `on_click`, 새 컴포넌트 등 자동 감지 |
| 스크린샷 과다 | 의미 있는 차이만 필터링, 동일 패턴 대표 1개만 유지 |
| 서버 기동/종료 관리 | stash 전후로 서버 재시작, 실패 시 자동 원복 |
| stash 충돌 | 충돌 감지 시 중단 + 사용자 알림 |

---

## 정리하면

이 스킬의 본질은 "스크린샷 자동화"가 아니에요.

**"프론트엔드 변경사항을 팀에 공유하는 전체 과정을 자동화한 것"**이에요.

코드 diff를 분석하고 → 영향받는 페이지를 찾고 → 변경 전후를 촬영하고 → 의미 있는 차이만 필터링하고 → 정리된 리포트를 생성하는 것. 이 전체 워크플로우를 한 줄 명령어로 실행할 수 있게 만든 거죠.

테스트 코드가 없는 프론트엔드에서, **시각적 변경사항을 추적하고 팀과 공유하는 구조**가 필요했고, 그 필요에서 이 도구가 나왔어요.
