## Claude가 만든 CSS를 열어봤더니

Overview 페이지는 파란색, 비교 페이지는 보라색. 카드 모서리가 어떤 건 둥글고, 어떤 건 각지고. 폰트 크기는 16px, 14px, 15px이 한 화면에 공존하고 있었어요. 20개 넘는 페이지를 이렇게 만들면, 각각 다른 사이트가 될 게 뻔했죠.

사내 퀀트 분석 도구의 리디자인이 필요했어요. v1은 기능 위주로 빠르게 만든 거라 페이지마다 스타일이 제각각이었거든요. "전체를 한 번 정리하자"고 마음먹고, [Claude Code](https://claude.ai/)한테 맡기면 빠르겠다 싶었어요.

> "Overview 페이지를 모던하게 바꿔줘. KPI 카드를 상단에 배치하고, 아래에 차트를 넣어줘."

이렇게 말로 시켰어요. 결과가 나오긴 해요. 근데 AI가 만든 티가 나더라고요. 색상 조합이 어색하고, 전체적으로 "그냥 돌아가기는 하는데 예쁘지는 않은" 느낌. 그리고 페이지를 하나 더 만들면 스타일이 완전히 달라지는 거예요.

**"알아서 잘 해줘"로는 안 되는구나.** 여기서 반나절 날렸어요.

---

## 첫 번째 시도: 페이지마다 프롬프트 쓰기

그래도 프롬프트를 더 구체적으로 쓰면 되지 않을까 싶었어요. "배경은 흰색, 카드는 둥글게, 폰트는 16px" 이런 식으로 페이지별로 세세하게 지시를 했죠.

아 근데 이게 왜 안 되냐면요. 매번 새 페이지를 시작할 때마다 AI가 이전 페이지에서 뭘 정했는지를 모르는 거예요. Overview에서 정한 색상 코드가 비교 페이지에서는 다른 값으로 바뀌어 있고, 간격도 24px이었다가 갑자기 20px이 되어 있고. 20개 페이지를 이렇게 하면 매번 수정의 수정이 필요한 거예요.

여기서 알게 된 건 — AI한테 일을 시킬 때 **"이번 페이지"가 아니라 "전체 시스템의 규칙"을 줘야 한다**는 거였어요.

## 그래서 HTML 목업을 직접 그렸어요

디자이너가 없으니까 제가 하기로 했어요. 피그마를 쓸 줄 모르니까, 가장 익숙한 방법으로 — HTML.

제일 복잡한 페이지부터 시작했어요. 백테스트 상세 페이지. KPI 카드 6개, 차트 6개, 테이블 2개, 히트맵까지 들어가는 무거운 페이지. 여기서 레이아웃과 색상, 폰트, 간격을 HTML/CSS로 직접 잡아봤어요.

이 과정에서 하나씩 디자인 결정을 내렸어요:

- **폰트**: Geist Sans. 숫자가 많은 대시보드라서 가독성 좋은 산세리프체를 골랐어요. 테이블의 숫자는 JetBrains Mono로 따로 지정했고요
- **색상**: Warm Zinc 톤 기반. 배경은 #f8f8f6 (살짝 따뜻한 회색), 카드는 흰색, 보더는 #e8e5e0. 수익률은 초록(#059669), 손실은 빨강(#dc2626), 강조는 파랑(#2563eb)
- **카드**: 16px 둥글기, 1px 보더, 살짝 그림자. 호버하면 그림자가 좀 더 진해지는 정도
- **간격**: 사이드바 220px, 탑바 44px, 콘텐츠 영역 좌우 32px 패딩

이걸로 기준이 잡히니까, 나머지 페이지 목업은 변형만 하면 됐어요. 결국 25개의 HTML 목업을 만들었어요.

## 드디어 게임 체인저: 디자인 토큰 문서화

목업을 만들고 나서 바로 AI에게 넘기면 될 줄 알았어요. "이 목업 보고 따라해줘."

근데 여기서 또 문제. AI가 새 페이지를 만들 때 이전 페이지 스타일을 기억하지 못하는 경우가 있었어요. 목업은 그림일 뿐이지, 규칙이 아니니까요.

그래서 디자인 토큰을 문서(DESIGN.md)로 정리했어요. 여기서 이틀 썼어요.

```
색상:
  배경: #f8f8f6
  카드: #ffffff
  보더: #e8e5e0
  텍스트 기본: #1a1a19
  텍스트 보조: #6b7280
  양수: #059669
  음수: #dc2626
  강조: #2563eb

타이포그래피:
  기본 폰트: Geist Sans
  숫자/코드: JetBrains Mono
  KPI 값: 26px, 700
  섹션 제목: 16px, 700
  뱃지: 11px, 600, uppercase

간격:
  카드 내부 패딩: 24px
  카드 간 간격: 20px
  카드 둥글기: 16px
```

이렇게 모든 규칙을 명시적으로 적어둔 거예요. AI에게 "이 문서를 참고해서 만들어줘"라고 하면, 어떤 페이지를 만들든 같은 스타일이 나오더라고요. 아까 반나절 삽질한 게 허무할 정도로.

근데 여기서 끝이 아니었어요. AI가 자주 빠지는 함정들이 있었거든요:

- "HTML을 넣을 때 반드시 `sanitize=False` 옵션을 써야 한다" — 안 쓰면 태그가 그대로 보여요
- "다크 모드 CSS 선택자는 `body--dark`이지 `body.dark`가 아니다" — 이거 때문에 다크 모드가 안 먹혀서 3시간 날렸어요
- "[Plotly](https://plotly.com/)에서 `add_vline` 쓰면 특정 조건에서 크래시 나니까 `shapes`를 써라"

이런 함정(gotchas)을 별도 문서로 정리해뒀어요. 같은 실수를 두 번 안 하려고.

## 실제 전환: 이건 생각보다 빨랐어요

기반이 잡히니까 페이지 전환은 꽤 빠르게 진행됐어요.

**진행 순서:**

1. **레이아웃 (layout_v2.py)** — 사이드바, 탑바, 다크 모드, CSS 변수. 모든 페이지가 공유하는 껍데기를 먼저 만들었어요
2. **Overview** — 가장 간단한 페이지부터 시작. KPI 카드 5개 + 차트 1개
3. **Backtest Result List** — AG Grid 테이블이 들어가는 복잡한 페이지. 여기서 테이블 스타일 기준을 잡았어요
4. **Backtest Detail** — 가장 무거운 페이지 (차트 6개, 테이블 2개). 이게 되면 나머지는 다 되는 거예요
5. **나머지 페이지들** — 비교, 상관관계, 리스크, 페이퍼 트레이딩, 데이터 소스 8개...

레이아웃과 디자인 토큰이 잡혀 있으니까, 후반부 페이지들은 "Overview의 KPI 카드 패턴으로 이 페이지도 만들어줘" 한마디면 됐어요.

## 근데 여기서 또 삽질이

순탄하지만은 않았어요.

**차트 라이브러리 호환 — 옵션 이름이 전부 다르다**

원래 Chart.js를 쓰고 있었는데, [NiceGUI](https://nicegui.io/)에서는 Plotly가 더 잘 맞더라고요. 근데 Chart.js 차트를 Plotly로 바꾸면 옵션 이름이 완전히 달라져요. `borderColor`가 `line_color`가 되고, `backgroundColor`가 `fillcolor`가 되고. 결국 변환 가이드를 별도 문서로 만들어서 AI에게 참고하게 했어요. 이 문서 만드는 데도 한나절.

**다크 모드에서 색이 깨지는 문제**

라이트 모드에서 잘 되던 게 다크 모드로 바꾸면 글자가 안 보이거나, 카드 배경이 이상해지거나. CSS 변수를 쓰는 건 알겠는데, 변수를 빠뜨린 곳이 하나라도 있으면 그 부분만 색이 깨지거든요. 20개 페이지에서 빠뜨린 변수를 찾아다니는 게 진짜... 결국 CSS 변수를 한 곳에서 관리하고, 모든 페이지가 반드시 그걸 참조하도록 구조를 다시 잡았어요.

**"거의 다 됐는데 이 부분만..."의 무한 반복**

AI가 90%를 만들어주면, 나머지 10%를 맞추는 데 체감 50%의 시간이 들었어요. 미세한 여백 3px 차이, 테이블 헤더 정렬, 반응형에서 카드가 깨지는 것. 이런 디테일은 결국 눈으로 직접 확인하고 피드백을 줘야 했죠. AI가 "됐습니다"라고 하는 걸 그대로 믿으면 안 돼요.

## 삽질에서 건진 것들

**AI는 "반복 작업"에 강해요.** 20개 페이지를 같은 디자인 시스템으로 전환하는 건, 패턴이 정해져 있고 규칙이 명확한 반복 작업이에요. 여기서 AI가 빠르고 정확하게 해줬어요. 반면 패턴을 정하는 것 — 어떤 색을 쓸지, 어떤 레이아웃이 이 데이터에 맞는지 — 이건 사람이 해야 했어요.

**"AI에게 잘 시키는 법"은 결국 "기획을 잘 하는 법"이었어요.** 이번 작업에서 가장 많은 시간을 쓴 건 코드 짜는 게 아니라, 목업을 만들고 디자인 규칙을 정리하는 거였어요. 프롬프트 엔지니어링이라고 하면 뭔가 기술적인 것 같지만, 실제로 해보면 **"요구사항을 명확하게 전달하는 능력"**이에요. 개발자한테 기획서를 전달할 때도 똑같은 원리죠.

솔직히, 전문 디자이너가 만든 것만큼 예쁘진 않아요. 반응형도 아직 모바일에서 좀 어색한 부분이 있고, 다크 모드에서 가끔 차트 범례 색이 배경에 묻히는 경우도 남아 있어요. 근데 사용자(리서처)들 반응은 좋았어요. 예쁘냐 아니냐보다 "일관되고 쓰기 편한가"가 더 중요했던 거죠.

## 다음에 같은 상황이면

비슷한 상황에서 AI로 대규모 리디자인을 한다면:

1. **말로 시키기 전에, 한 페이지라도 직접 목업을 만들어라.** 가장 복잡한 페이지부터. 거기서 기준이 나온다
2. **디자인 토큰을 문서로 먼저 정리해라.** 색상, 폰트, 간격, 둥글기 전부. AI가 매번 참조할 수 있게. 이걸 빼먹으면 페이지마다 다른 사이트가 나온다
3. **AI가 자주 실수하는 것들(gotchas)을 별도로 모아라.** 같은 실수를 두 번 하면 그건 AI 탓이 아니라 문서 탓이다
4. **90%에서 멈추지 마라.** AI가 "완료"라고 해도 직접 눈으로 확인해라. 마지막 10%가 품질을 결정한다
5. **한 페이지에서 검증하고, 그 다음에 전체에 적용해라.** 20개를 한꺼번에 바꾸면 수정 범위가 감당이 안 된다
