# 포트폴리오 블로그 글쓰기 가이드

## 톤 & 문체

- **~했어요, ~거든요, ~했죠** 체를 기본으로 사용
- 구어체와 문어체를 섞되, 구어체 비중을 7:3 정도로
- 감정을 자연스럽게 넣기: "솔직히", "꽤", "뭔가…", "근데"
- 짧은 문장과 긴 문장을 번갈아 써서 리듬감 만들기
- 독자에게 질문 던지기: "~하면 되는 걸까요?", "~할까요?"
- 교훈을 선언하지 않고, 과정을 보여주면서 깨달음이 드러나게

## 글 구조

### 도입 (상황 설명)
- 팀 구성, 환경, 제약 조건을 먼저 깔기
- "문제는 ~였다"로 핵심 갈등 드러내기
- 1인칭 시점에서 자연스럽게 시작

### 전개 (문제 → 시도 → 해결)
- 처음 시도한 것 + 왜 안 됐는지
- 실제로 만든 것 + 왜 이걸 선택했는지
- 구체적인 디테일 (기능 설명, 수치, 기술 선택 이유)
- before/after, 비교 대상이 있으면 활용

### 마무리 (배운 것)
- "돌아보면", "느낀 것" 같은 자연스러운 전환
- 교훈을 3개 이내로
- 각 교훈은 볼드 한 줄 + 2~3줄 설명
- 다음 글 예고 (시리즈일 때)

## 강조 표현 규칙

- **볼드**: 핵심 메시지, 깨달음, 반전 포인트에만 사용
- `코드 백틱`: 기술 용어, 기능 이름, 태그 이름 등
- > 인용: 사용자(리서처)의 실제 말, 프롬프트 예시
- 목록(불릿): 기능 나열, 순서 있는 과정 설명
- 표: 비교(before/after, 옵션 비교)에 활용
- `<!-- 코멘트 -->`: 스크린샷 삽입 위치 표시

## 피해야 할 것

- 논문체 (~이다, ~하였다, ~것이다)
- 과도한 기술 용어 나열 (설명 없이 던지지 않기)
- 자기 자랑 톤 ("저는 ~를 해냈습니다")
- 추상적인 교훈 ("소통이 중요합니다")
- 이모지 사용

## 기획자/PM 포지션을 위한 관점

글 전체에서 드러나야 하는 것:

1. **사용자 관점**: "사용자가 뭘 원하는지"를 먼저 생각하는 사람
2. **문제 정의 능력**: 표면적 요청 뒤에 있는 진짜 니즈를 찾는 과정
3. **기술 이해**: 기술을 이해하면서 의사결정을 내리는 모습
4. **AI 활용**: AI를 도구로 활용하되, 판단은 사람이 한다는 인식
5. **실행력**: 리소스 부족 속에서 직접 만들어낸 경험

기술 설명을 할 때는 "이걸 왜 선택했는가" 관점으로. 코드 자체가 아니라 의사결정을 보여주기.

## Astro 블로그 프론트매터

```yaml
---
title: "제목 — 부제가 있으면 em dash(—)로 연결"
description: "1~2줄 요약. 검색/미리보기에 노출되는 텍스트"
date: YYYY-MM-DD
tags: ["Portfolio", "AI", "서비스기획"]
category: "project"
featured: true/false
---
```

## 섹션 구분

- `##`으로 큰 섹션 나누기
- `###`으로 세부 항목 (사례 1, 사례 2 등)
- 섹션 사이에 `---` (수평선)으로 시각적 구분
- 섹션 제목은 짧고 직관적으로 ("삽질한 것들", "근데, 프론트를 어떻게?")
