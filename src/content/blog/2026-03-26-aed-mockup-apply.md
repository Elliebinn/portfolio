## 60번. 검수-피드백을 반복한 횟수예요.

20페이지를 목업대로 바꾸면서 세어봤어요. 페이지당 평균 3번의 수정 사이클. 목업 넘기고 → AI가 코드 뱉고 → 브라우저 열어서 확인하고 → "여기 다르다" 피드백 → 수정 → 다시 확인. 이걸 20페이지에 걸쳐 반복하니까 60번이더라고요.

구현보다 검수에 시간을 더 쓰고 있었어요.

---

## 목업을 넘기면 그대로 나올 줄 알았어요

[HTML 목업 25개를 직접 만들고](/ko/blog/2026-03-23-aed-ai-redesign-chaos/), 디자인 토큰(DESIGN.md)까지 정리했어요. 색상, 폰트, 간격 규칙을 문서화해서 AI에게 "이거 보고 구현해줘"라고 넘기면 될 줄 알았죠.

근데 실제로 해보면 거의 매번 이런 상황이 벌어져요.

- KPI 카드 간격이 목업이랑 미묘하게 달라요
- 테이블 헤더 스타일이 DESIGN.md 규칙을 안 따르고 인라인으로 들어가 있어요
- 차트 색상이 목업에서는 `#059669`인데 구현에서는 비슷하지만 다른 초록색이에요
- 목업에 있던 호버 효과가 빠져 있어요

하나하나 잡아서 "여기 고쳐줘" 하고, 고쳐지면 다시 확인하고, 또 다른 게 눈에 들어오고. 이 사이클이 한 페이지당 서너 번.

디자인 차이는 그래도 눈에 보이니까 잡을 수 있었어요. 진짜 문제는 따로 있었어요.

---

## 기존에 잘 되던 기능이 사라져요

목업은 정적인 HTML이에요. 다크 모드 토글, 한/영 전환, 데이터 로딩 로직, 이벤트 핸들러 같은 건 목업에 표현할 수가 없어요. 근데 AI한테 "이 목업대로 만들어줘"라고 하면, **목업에 없는 건 코드에서도 빼버리는** 경우가 있었어요.

실제로 겪은 일들:

- 사이드바 접기/펼치기 토글이 사라짐
- 다크 모드 전환 시 CSS 변수를 안 써서 색이 깨짐
- 테이블 정렬 클릭 이벤트가 없어짐
- API 호출 로직이 더미 데이터로 대체됨

기본 화면은 멀쩡해 보여요. 근데 클릭하거나 모드를 바꾸면 터지는 거예요. 나중에 발견하면 원인 찾는 것도 더 오래 걸리고요.

20페이지에서 같은 종류의 삽질을 반복하고 있었어요. "이번에도 다크 모드가 빠졌네", "이번에도 이벤트 핸들러가 없어졌네" — 이 패턴을 보면서, 이건 내가 수동으로 잡을 문제가 아니라 **프로세스 자체를 바꿔야 하는 문제**라는 걸 알았어요.

---

## 원인: "뭘 할지" 정의 없이 바로 구현에 들어갔어요

돌아보면, 문제는 두 가지였어요.

1. **목업과 코드를 비교하는 과정에 체계가 없었어요** — 뭘 추가하고, 뭘 바꾸고, 뭘 건드리면 안 되는지를 사전에 정리 안 하고 바로 구현에 들어가니까 빠지는 게 생기는 거예요
2. **"건드리면 안 되는 것"의 목록이 없었어요** — AI는 목업에 없으면 필요 없다고 판단할 수 있어요. "이건 절대 빼지 마"를 명시적으로 말해줘야 하는데, 그걸 매번 기억에 의존하고 있었어요

그래서 만든 게 [mockup-apply](https://github.com/anthropics/claude-code) 스킬이에요.

---

## 핵심: ADD / CHANGE / KEEP

구현 시작 전에, 모든 요소를 세 카테고리로 분류해요.

| 분류 | 의미 | 예시 |
|------|------|------|
| **ADD** | 목업에 있고, 코드에 없는 것 | 새 KPI 카드, 태그 필터 칩 |
| **CHANGE** | 양쪽 다 있지만 다른 것 | 카드 레이아웃 변경, 색상 수정 |
| **KEEP** | 코드에 있고, 목업에 없는 것 | 다크 모드, API 로직, 이벤트 핸들러 |

특히 **KEEP이 게임 체인저**였어요. 목업에 없더라도 "이건 절대 빼면 안 된다"는 목록을 미리 만들어두면, 작업하면서 실수로 날려버리는 걸 막을 수 있거든요.

---

## 사용법: 파일 하나 넘기면 4단계 자동 실행

```
/mockup-apply mockup_backtest_detail.html
```

### Phase 1: 분석

목업 HTML과 현재 코드를 양쪽 다 읽어요. 목업에서는 레이아웃, 컴포넌트, 스타일, 인터랙션을 파악하고, 코드에서는 데이터 로딩 로직, 이벤트 핸들러, API 엔드포인트를 파악해요. 그리고 ADD/CHANGE/KEEP 체크리스트를 생성해서 보여줘요.

핵심은 **승인 없이 구현을 시작하지 않는다**는 거예요. 체크리스트를 보고 "이건 CHANGE가 아니라 KEEP이야" 같은 피드백을 줄 수 있어요.

### Phase 2: 디자인 시스템 동기화

목업 CSS에서 새 디자인 토큰을 추출하고, DESIGN.md와 비교해요. 이미 정의된 토큰은 그대로 쓰고, 새 토큰은 추가하고, 충돌하면 어느 쪽을 따를지 물어봐요. 이렇게 하면 "이 페이지만 색이 다른" 문제가 안 생겨요.

### Phase 3: 구현

체크리스트 순서대로 하나씩 구현해요. ADD 먼저, CHANGE 다음. 한꺼번에 전부 바꾸지 않아요. 그리고 KEEP 목록에 있는 건 절대 건드리지 않아요.

### Phase 4: 검증

서버를 띄워서 페이지를 열고, 스크린샷을 찍고, 콘솔 에러를 체크해요. 그리고 KEEP 항목들이 전부 동작하는지 확인해요. 다크 모드 토글이 동작하는지, 데이터 로딩이 실제 데이터를 가져오는지, 이런 것들을.

## Before / After

| | Before (수동) | After (mockup-apply) |
|---|---|---|
| 시작 | 목업 넘기고 바로 구현 시작 | 분석 → 체크리스트 → 승인 → 구현 |
| 기존 기능 보존 | AI 재량 (종종 빠짐) | KEEP 목록으로 명시적 보호 |
| 디자인 일관성 | 페이지마다 미세하게 다름 | DESIGN.md 동기화로 토큰 통일 |
| 검수 횟수 | 페이지당 3~4회 수정 사이클 | 체크리스트로 1회에 대부분 해결 |
| 검증 | 수동으로 브라우저 확인 | 자동 스크린샷 + 콘솔 에러 체크 |

검수 횟수가 3~4회에서 거의 1회로 줄었어요. 체감 차이가 큰데, 수치로 보면 더 명확해요. 20페이지 기준 60번이던 검수 사이클이 20번 남짓으로 줄어든 거니까요. 그 시간에 다음 페이지 작업을 할 수 있게 됐어요.

---

## 이런 상황에서 쓰면 좋아요

- **HTML/Figma 목업을 코드로 전환**할 때 디테일이 자꾸 빠지는 경우
- 기존 코드에 **동적 기능(토글, API 호출, 이벤트)**이 많아서 목업 적용 시 깨지는 경우
- 디자인 시스템이 있는데 **페이지마다 미세하게 달라지는** 경우
- AI에게 구현을 맡기는데 **"알아서 해줘"가 잘 안 되는** 경우

## 스킬로 독립시켰어요

20페이지를 거치면서 패턴이 충분히 검증됐다고 판단해서, 프로젝트에 묶여 있던 워크플로우를 **독립된 Claude Code 스킬**로 분리했어요. 다른 프로젝트에서도 `/mockup-apply` 한 줄이면 같은 파이프라인이 돌아가요.

```bash
git clone https://github.com/Elliebinn/mockup-apply.git ~/.claude/skills/mockup-apply
```

가이드 문서도 만들었어요. 예시 3개 — 대시보드 리디자인, 분석만 수행(`--analyze-only`), AI 목업 생성 후 바로 적용하는 파이프라인 — 를 넣어서 처음 쓰는 사람도 흐름을 잡을 수 있게요.

[frontend-checkpoint](/ko/blog/2026-03-23-aed-frontend-checkpoint/)가 "변경 후 기록"을 자동화한 도구라면, mockup-apply는 **"변경 자체"를 체계화한 도구**예요. 둘을 같이 쓰면 목업 적용 → 체크포인트 촬영까지 한 흐름으로 이어져요.

다음에는 KEEP 항목들에 대한 자동 회귀 테스트를 붙일 생각이에요. 지금은 Phase 4에서 한 번 검증하는데, 이후 다른 페이지 작업하면서 이전 KEEP 항목이 다시 깨질 수도 있거든요. 전체 KEEP 목록을 모아서 PR 단위로 한 번에 검증하는 걸 만들려고요.
