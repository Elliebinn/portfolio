## 사용자가 "태그 추가해주세요"라고 했을 때, 그걸 그대로 만들면 될까요?

사내 분석 도구를 운영하면서 리서처들의 피드백을 자주 받았어요. "이거 불편해요", "이 기능 추가해주세요" 같은 요청이 꾸준히 들어왔죠.

여기서 선택지가 갈렸어요. **요청을 그대로 구현하는 것과, 요청 뒤에 숨은 진짜 문제를 찾는 것.** 같은 시간을 써도 결과의 깊이가 완전히 달라지는 갈림길이에요.

가설은 이랬어요: 사용자가 말하는 "불편함"은 증상이지, 진단이 아니다. 표면적 요청을 한 층 아래로 파고들어야 진짜 기획이 시작된다.

두 가지 사례로 이 가설을 검증해봤어요.

---

## 사례 1: "전략이 너무 많아서 못 찾겠어요"

### 표면 요청

사내 분석 도구의 백테스트 비교 페이지 — 여러 전략의 수익률을 한 화면에 올려놓고 비교하는 핵심 기능이에요. 리서처들이 거의 매일 쓰는 페이지죠.

전략이 점점 늘어나 200개를 넘어가면서, 이런 피드백이 들어왔어요.

> "내가 지난주에 돌린 전략 찾으려면 한참 스크롤해야 해"
> "비슷한 전략끼리 묶어서 보고 싶은데 방법이 없어"

### 가설 수립

표면적 요청은 "검색 기능을 넣어달라"에 가까웠어요. 여기서 한 단계 더 들어가봤어요.

리서처들이 전략 이름을 정확히 기억하고 있을까요? 자동 생성된 ID 같은 이름이 대부분이에요. 그리고 "비슷한 전략끼리 묶어서 보고 싶다"는 요청은 검색으로 해결이 안 돼요.

**가설: "못 찾겠다"의 본질은 "내 기준으로 분류하고 싶다"이다.**

### 선택지 비교

| 옵션 | 장점 | 한계 |
|------|------|------|
| **검색 기능** | 구현 간단, 이름을 아는 경우 빠름 | 이름을 모르면 무용, 분류 불가 |
| **폴더 구조 (트리형)** | 계층적 정리 가능 | 전략 하나가 여러 카테고리에 속할 때 한계 |
| **태그 시스템** | 다중 분류, 사용자 정의 가능 | 태그 관리 필요, 초기 학습 비용 |

핵심 트레이드오프는 **구조의 유연성**이었어요. "모멘텀"이면서 동시에 "한국주식"인 전략이 있는데, 폴더는 하나에만 넣을 수 있지만 태그는 여러 개 붙일 수 있죠.

### 의사결정: 태그 시스템

태그 기반 분류 체계를 선택한 결정적 이유는 두 가지였어요.

첫째, **다중 분류**. 전략의 속성은 단일 축으로 나눌 수 없어요. 유형(모멘텀/밸류), 자산군(한국주식/미국ETF), 실험 시기 — 이 세 축이 동시에 필요했어요.

둘째, **사용자 자율성**. 미리 정해진 카테고리에 끼워 맞추는 게 아니라, 리서처 본인이 자기 기준으로 분류할 수 있어야 했어요. 실제로 어떤 사람은 전략 유형으로, 어떤 사람은 실험 시기로 태그를 붙이더라고요.

구체적으로 이렇게 설계했어요:

- 비교 페이지 검색창 옆에 태그 필(pill) 버튼 배치. `모멘텀 (12)` 형태로 태그 이름과 해당 전략 수를 함께 표시
- 태그 클릭 시 해당 전략만 필터링, 다시 누르면 해제
- 태그 활성 상태에서 검색하면 **태그 범위 안에서만** 검색 — 범위를 좁혀가는 흐름
- 검색 결과와 성과 비교 테이블에서 각 전략의 태그를 뱃지로 표시

### 왜 그 결론을 냈는지

태그를 선택한 건 "태그가 멋져 보여서"가 아니라 **도메인 특성** 때문이었어요.

퀀트 전략은 단일 축으로 분류되지 않아요. "모멘텀"이면서 "한국주식"이고, 동시에 "2025년 하반기 실험"인 전략이 있거든요. 폴더(트리형)를 썼다면 이 전략을 어디에 넣어야 할지 매번 고민해야 했을 거예요. 태그는 이 문제를 구조적으로 피해가요.

두 번째 판단 근거는 **사용자 자율성**이었어요. 분류 기준이 사람마다 달랐어요. 어떤 리서처는 전략 유형으로, 어떤 리서처는 실험 시기로 묶고 싶어했어요. 미리 정해진 카테고리에 끼워 맞추게 하면, 결국 "내 기준"이 아니라 "시스템의 기준"으로 분류하게 돼요. 태그는 각자 자기만의 분류 체계를 만들 수 있게 해줘요.

폴더 방식은 구현이 더 간단했어요. 그럼에도 태그를 선택한 건, 단기 구현 비용보다 장기 사용성이 더 중요했기 때문이에요.

### 검증

**"못 찾겠다"는 불만의 본질은 "내 기준으로 분류하고 싶다"였다** — 이 가설이 맞았어요. 검색 기능만 넣었다면 해결되지 않았을 문제를, 태그 시스템이 해결했죠.

## 사례 2: "날짜 선택이 불편해요"

### 표면 요청

백테스트 상세 페이지의 Holdings 섹션 — 특정 날짜에 포트폴리오가 어떤 종목을 얼마나 들고 있었는지 보여주는 테이블이에요.

기존에는 날짜를 드래그 슬라이더로 선택하게 되어 있었어요. 리서처 한 분의 피드백:

> "날짜를 드래그로 선택하는데, 내가 보고 싶은 날짜로 정확히 가기가 힘들어"

### 가설 수립

표면적 해결책은 드래그를 date picker로 바꾸는 거예요. 정확한 날짜 지정은 되겠죠.

여기서 도메인 경험이 판단을 바꿨어요. 저는 모회사인 에셋플러스자산운용에서 실제로 AI 공모펀드를 운용한 경험이 있어요. 그 경험에서 알고 있었던 것 — **실무에서 포트폴리오를 볼 때 가장 중요한 건 리밸런싱 시점이에요.**

리밸런싱은 포트폴리오 종목 구성을 바꾸는 날이에요. 이 날 전후로 어떤 종목이 들어오고 나갔는지, 비중이 어떻게 변했는지를 추적하는 게 핵심이죠. 수백 개의 날짜 중에서 리서처가 진짜 보고 싶은 건 리밸런싱 날짜예요.

**가설: "날짜 선택이 불편하다"의 본질은 "리밸런싱 전후 포지션 변화를 쉽게 추적하고 싶다"이다.**

### 선택지 비교

| 옵션 | 해결하는 것 | 해결하지 못하는 것 |
|------|------------|-------------------|
| **Date picker로 교체** | 정확한 날짜 지정 | 어떤 날짜를 봐야 하는지 모름 |
| **캘린더 + 리밸런싱 표시** | 날짜 지정 + 중요 날짜 인지 | 빠른 이동 불가 |
| **리밸런싱 포인트 네비게이션** | 핵심 날짜 간 빠른 이동 + 맥락 파악 | 구현 복잡도 높음 |

### 의사결정: 리밸런싱 포인트 네비게이션

날짜 선택 UI를 완전히 새로 설계하는 선택을 했어요. 구현 복잡도는 높지만, 리서처의 실제 워크플로우에 맞는 도구를 제공하는 것이 더 높은 가치라고 판단했어요.

- **일별 이동 버튼**: 하루씩 앞뒤로 이동
- **날짜 직접 입력**: date picker로 원하는 날짜 지정
- **리밸런싱 포인트 이동**: 이전/다음 리밸런싱 날짜로 바로 점프. 노란색 버튼으로 시각적 구분
- **리밸런싱 표시**: 현재 선택 날짜가 리밸런싱일이면 주황색 점 표시
- **최신 이동 버튼**: 가장 최근 날짜로 한 번에 이동

<!-- Holdings 날짜 툴바 스크린샷 -->

여기서 한 단계 더 나갔어요. Holdings 테이블에서 종목을 클릭하면 위에 있는 포트폴리오 비중 차트에 그 종목의 비중 변화가 오버레이로 표시돼요. 최대 5개까지 겹쳐볼 수 있어서, "이 종목의 비중이 시간이 지나면서 어떻게 변했는지"를 차트와 테이블을 오가면서 분석할 수 있죠.

<!-- 오버레이 기능 스크린샷 -->

이 오버레이 기능은 요청이 들어온 적 없어요. 펀드 운용 경험에서 나온 판단이었어요 — 실무에서 자연스럽게 하는 행동이 "이 종목 비중이 왜 이렇게 변했지?" 하고 추적하는 거거든요.

### 왜 그 결론을 냈는지

리밸런싱 포인트 네비게이션을 만들기로 한 결정은, 솔직히 구현 복잡도를 감수한 선택이었어요. 날짜 선택 달력(date picker)으로 바꾸는 건 하루면 됐어요. 리밸런싱 포인트를 실시간으로 계산해서 "이전/다음 리밸런싱 날짜"를 알려주려면 훨씬 복잡했죠.

그럼에도 복잡한 방향을 택한 근거는 두 가지였어요.

첫째, **date picker는 "어떤 날짜를 봐야 하는지"를 해결하지 못해요.** 달력에서 날짜를 직접 고르려면 리서처가 "리밸런싱이 몇 월 며칠이었지?"를 기억하고 있어야 해요. 수백 개 전략의 리밸런싱 날짜를 외우고 있는 사람은 없어요. UI를 예쁘게 바꿔봤자 문제의 본질은 그대로인 거예요.

둘째, **실무에서 포트폴리오를 보는 방식**이 있어요. 펀드 운용을 해본 사람이라면 안다는 것 — 포트폴리오를 추적할 때 임의의 날짜가 아니라 "리밸런싱 전날 vs 리밸런싱 다음날"을 보거든요. 포지션 변화의 맥락이 그 안에 있으니까요. 이 행동 패턴을 UI로 옮긴 게 리밸런싱 포인트 네비게이션이에요.

오버레이 기능은 이 연장선에 있었어요. 리밸런싱 날짜를 이동하면서 "이 종목이 이번 리밸런싱에서 들어왔구나"를 알게 되면, 자연스럽게 다음 질문은 "그 종목의 비중이 시간이 지나면서 어떻게 변했는지"예요. 그 질문을 예상하고 기능을 만들어뒀어요.

### 검증

리서처들이 "이런 것도 되네?"라고 반응한 지점이 바로 오버레이 기능이었어요. 요청한 적 없는 기능인데 써보니까 편하다는 거죠. 이런 경험이 쌓이면 "이 사람이 만들면 쓸만한 게 나온다"는 신뢰가 생기고, 그 이후로는 더 솔직한 피드백이 들어오기 시작했어요.

## 프레임워크: 사용자 요청의 세 층 분석

두 사례를 통해 하나의 판단 기준이 만들어졌어요. 이후로 사용자 요청을 받으면 이 프레임워크로 분해해요.

| 층 | 설명 | 사례 1 (태그) | 사례 2 (네비게이션) |
|----|------|--------------|-------------------|
| **표면 요청** | 사용자가 말한 것 | "검색 기능 넣어주세요" | "날짜 드래그가 불편해요" |
| **실제 행동** | 사용자가 하려는 것 | 비슷한 전략끼리 묶어서 보기 | 특정 시점의 포지션 변화 추적 |
| **근본 니즈** | 해결하면 만족하는 것 | 내 기준으로 자유롭게 분류하기 | 리밸런싱 전후 비교 분석 |

**표면 요청에서 멈추면 검색창을 넣고, 실제 행동까지 파고들면 태그 시스템을 설계하고, 근본 니즈까지 도달하면 리밸런싱 네비게이션과 종목 오버레이까지 나와요.**

여기에 한 가지 변수가 더 있어요 — **도메인 경험**. 같은 "불편하다"를 듣더라도, 그 업무를 해본 사람과 안 해본 사람은 물을 수 있는 질문의 깊이가 달라요. "검색을 넣을까요?"가 아니라 "어떤 기준으로 전략을 분류하세요?"라고 물을 수 있는 것. 도메인 이해가 깊을수록 세 번째 층에 더 빨리 도달할 수 있어요.

### 이 프레임워크를 다른 상황에서 쓰는 법

프레임워크를 재사용하려면, 각 층에서 어떤 질문을 던질지 알아야 해요.

**표면 요청을 들었을 때:**
- "그걸 구현하면 지금 불편함이 사라지나?"
- "이 요청을 그대로 만들면, 사용자가 실제로 하려는 걸 할 수 있나?"

**실제 행동을 파악할 때:**
- "이 기능을 쓰는 사람이 직전에 무슨 행동을 했고, 직후에 무슨 행동을 하려고 하는가?"
- 사용자를 직접 관찰하거나, 도메인 경험으로 그 행동 흐름을 그려볼 수 있어야 해요.

**근본 니즈까지 파고들 때:**
- "이 기능이 없으면 어떤 목표를 달성하지 못하는가?"
- 이 질문에 답할 수 있으면, 표면 요청과 전혀 다른 해결책이 보이기 시작해요.

어떤 요청이든 이 세 층을 명시적으로 분리해서 정리하는 것만으로도, "그냥 구현하기"와 "제대로 설계하기"의 차이가 나요.
