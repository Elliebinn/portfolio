## 코드 수정 5분. 공유 준비 20분.

코드를 고치는 건 빨라요. 사이드바 active 색상 바꾸기, 5분. 차트 레이블 정렬 맞추기, 3분. 근데 그 다음이 문제예요.

바꾸기 전 스크린샷 찍기. 바꾼 후 스크린샷 찍기. "여기가 달라졌어요" 텍스트 정리하기. 스크린샷이랑 같이 클릭업에 올리기. **매번 20분.** 이걸 하루에 서너 번 하면 코드 수정보다 공유 준비에 시간을 더 쓰고 있었어요.

그리고 솔직히, 바빠지면 "바꾸기 전 스크린샷"을 깜빡해요. 그러면 before/after 비교 자체가 불가능해지는 거죠. 이전 화면이 어땠는지 기억에 의존하게 되는데, "여백이 좀 더 넓었나? 글자 크기가 달랐나?" 이런 건 절대 정확하게 떠올릴 수 없어요.

---

## 왜 프론트엔드는 유독 이런 문제가 있을까

백엔드는 테스트를 돌리면 돼요. 코드 바꾸고, 테스트 실행하고, 통과하면 끝.

근데 우리 분석 도구는 [NiceGUI](https://nicegui.io/)라는 Python 기반 웹 프레임워크로 만들어져 있어요. UI 쪽에 유닛 테스트를 넣을 수 있는 구조가 아니었어요. 변경을 확인하는 방법이 **브라우저에서 직접 눈으로 보는 것** 딱 하나였죠.

거기에 20개 넘는 페이지를 리디자인하면서, 한 번 수정으로 여러 페이지가 영향 받는 경우가 많았어요. 사이드바 스타일을 바꾸면 모든 페이지의 사이드바가 바뀌잖아요. 어떤 페이지를 대표로 캡처해야 하는지도 매번 고민이었고요.

더 골치 아픈 건 인터랙션이 있는 변경이에요. "차트를 클릭하면 오버레이가 뜨는 기능"을 추가했다고 해볼게요. 기본 상태 스크린샷만으로는 이 변경이 안 보여요. 실제로 클릭해서 오버레이가 뜬 상태를 캡처해야 하는데, 이런 시나리오를 매번 수동으로 재현하고 있었어요.

---

## "이거 자동화해야겠다"

스크린샷 찍다가 문득 든 생각이, **git이 이전 상태를 이미 갖고 있잖아** 였어요.

`git stash`로 수정 전 상태를 복원해서 스크린샷을 찍고, `git stash pop`으로 수정 후 상태를 적용해서 같은 화면을 다시 찍으면 — "깜빡하고 before를 안 찍었다" 문제가 사라지는 거죠. 게다가 `git diff`를 분석하면 어떤 페이지가 영향을 받는지도 자동으로 알 수 있고요.

그래서 만든 게 `/frontend-checkpoint` 스킬이에요.

---

## 사용법: 명령어 한 줄

```
/frontend-checkpoint "사이드바 active 하이라이트 수정"
```

이 한 줄이면 끝이에요. 나머지는 자동으로 돌아가요.

1. `git diff`로 변경 파일 분석 → `@ui.page` 데코레이터와 import 관계를 추적해서 영향받는 페이지 라우트 파악
2. `git stash`로 수정 전 상태 복원 → 스크린샷 촬영
3. `git stash pop`으로 수정 후 상태 적용 → 같은 화면 다시 촬영
4. before/after 비교 → 의미 있는 차이만 필터링
5. 리포트 자동 생성

결과물은 이렇게 생겨요.

```
.screenshots/checkpoints/
└── 2026-03-23_web-redesign_title-sidebar-unify/
    ├── REPORT.md          <- 변경 요약 + before/after 비교
    ├── meta.json          <- 변경 파일, 브랜치, 커밋 등 메타데이터
    ├── overview_before.png
    ├── overview_after.png
    ├── result-list_before.png
    ├── result-list_after.png
    └── ...
```

REPORT.md 안에는 변경 요약, 파일별 상세 diff, Before/After 스크린샷 비교, 제외된 시나리오 목록까지 다 들어 있어요. 이걸 그대로 클릭업에 올리면 끝.

---

## 인터랙션 캡처가 핵심이었어요

기본 상태 스크린샷만 찍으면 못 잡는 변경이 너무 많았어요. 그래서 **동적 시나리오 탐색** 기능을 넣었어요.

[Playwright](https://playwright.dev/)로 페이지를 열고, DOM 구조를 분석해서 클릭할 수 있는 요소(탭, 버튼, 드롭다운, 테이블 행)를 자동으로 찾아요. 하나씩 클릭하면서 상태가 변하면 스크린샷을 찍는 거죠.

더 중요한 건, `git diff`를 분석해서 **새로 추가된 기능**을 찾아내는 거였어요. `on_click`이 추가됐으면 그 요소를 클릭하는 시나리오를, 새 차트가 추가됐으면 그 차트 영역을 캡처하는 시나리오를 자동으로 만들어요.

시나리오가 자동으로 생성되다 보면 before/after 쌍이 엄청 많아지는데, 그중 90%는 변화가 없거나 다른 시나리오랑 같은 변화예요. 그래서 필터링 단계를 넣어서 의미 있는 차이만 남기고, 비슷한 패턴은 대표 하나만 유지해요.

---

## Before / After

| | Before (수동) | After (체크포인트) |
|---|---|---|
| 변경 전 기록 | 기억나면 스크린샷 (자주 까먹음) | git stash 기반 자동 복원 |
| 영향 범위 파악 | "이 파일 바꾸면 어디 영향 가지?" 고민 | import 관계 추적으로 자동 탐지 |
| 인터랙션 변경 | 수동으로 클릭해서 캡처 | DOM 분석 → 자동 시나리오 생성 |
| 리포트 작성 | 스크린샷 + 텍스트 정리에 20분 | 자동 생성, 0분 |
| 공유 주기 | 귀찮아서 변경 몇 개 모아서 한 번에 | 변경할 때마다 바로 |

체감상 가장 큰 변화는 공유 빈도였어요. 전에는 귀찮아서 변경을 모아두다가 한 번에 올렸는데, 지금은 수정하고 바로 체크포인트 돌리니까 팀원들이 변경사항을 실시간으로 확인할 수 있게 됐어요.

---

## 이런 상황에서 쓰면 좋아요

- **테스트 코드 없는 프론트엔드**에서 시각적 변경을 추적해야 할 때
- 한 번 수정으로 **여러 페이지에 영향**이 가는 변경을 할 때
- 변경사항을 비개발자 팀원에게 **시각적으로 공유**해야 할 때
- 인터랙션이 포함된 변경을 **기본 스크린샷만으로 보여주기 어려울 때**

---

## 기술적으로 풀어야 했던 것들

| 문제 | 해결 |
|------|------|
| 변경 파일 → 영향 페이지 매핑 | `@ui.page` 데코레이터 + import 관계 추적 |
| 변경 전 상태 복원 | `git stash` / `git stash pop` 활용 |
| 인터랙션 캡처 | Playwright DOM 스냅샷 분석 → 클릭 가능 요소 자동 탐색 |
| 새 기능 누락 방지 | `git diff`에서 `on_click`, 새 컴포넌트 등 자동 감지 |
| 스크린샷 과다 | 의미 있는 차이만 필터링, 동일 패턴 대표 1개만 유지 |
| 서버 기동/종료 관리 | stash 전후로 서버 재시작, 실패 시 자동 원복 |
| stash 충돌 | 충돌 감지 시 중단 + 사용자 알림 |

## 현재 v1. 다음은 목업 대조 검증

지금은 "변경 전후를 자동으로 기록하고 리포트를 생성하는" 수준이에요. 잘 동작하고 있고, 실제로 사이드바 하이라이트를 고쳤을 때 "10개 파일 수정, 7개 페이지 영향, 대표 3개 페이지 before/after"가 깔끔하게 나왔어요. 수동이었으면 30분은 걸렸을 작업이 명령어 한 줄로 끝났고요.

근데 한계가 하나 있어요. 이 도구는 "있는 것이 제대로 동작하는가"만 검증해요. "있어야 하는데 없는 것"은 못 찾아요. 목업이라는 정답지와 대조하는 단계가 아직 없거든요.
