## 해야 할 일은 명확했어요

지난 글들에서 얘기한 것처럼, 저는 사내 퀀트 분석 도구에서 사용자 니즈를 파악하고 기능을 기획하는 역할을 하고 있었어요. 태그 시스템도 만들었고, Holdings 네비게이션도 개선했고요.

근데 기능을 하나씩 추가하다 보니까 화면이 점점 지저분해지는 거예요. v1은 기능 위주로 빠르게 만든 거라, 페이지마다 스타일이 다르고, 여백도 들쭉날쭉하고, 전체적으로 통일감이 없었어요.

"전체를 한 번 깔끔하게 정리해야겠다"는 생각은 계속 있었는데… 20개가 넘는 페이지를 어떻게요? 디자이너도 없고, 프론트엔드 개발자도 없는데?

---

## 1단계: 말로 시키기 — 그리고 실패

처음에는 Claude Code에 말로 설명했어요.

> "Overview 페이지를 모던하게 바꿔줘. KPI 카드를 상단에 배치하고, 아래에 차트를 넣어줘."

결과가 나오긴 해요. 근데 뭔가… AI가 만든 티가 나더라고요. 색상 조합이 어색하고, 폰트 크기가 들쭉날쭉하고, 전체적으로 "그냥 돌아가기는 하는데 예쁘지는 않은" 느낌이었어요.

그리고 더 큰 문제가 있었어요. 페이지를 하나 더 만들면 스타일이 완전히 달라지는 거예요. Overview는 파란색 계열인데, 비교 페이지는 갑자기 보라색이 나오고, 카드 모서리도 어떤 건 둥글고 어떤 건 각지고. 20페이지를 이렇게 만들면 각각 다른 사이트처럼 보일 게 뻔했죠.

**AI한테 "알아서 잘 해줘"는 안 되는구나**를 여기서 배웠어요.

---

## 2단계: HTML 목업을 직접 그렸어요

디자이너가 없으니까 제가 직접 하기로 했어요. 근데 피그마 같은 디자인 도구를 쓸 줄 모르니까, 가장 익숙한 방법으로 — HTML.

백테스트 상세 페이지부터 시작했어요. 이게 제일 복잡한 페이지거든요. KPI 카드 6개, 차트 6개, 테이블 2개, 히트맵까지 들어가는. 이 페이지의 레이아웃과 색상, 폰트, 간격을 HTML/CSS로 직접 잡아봤어요.

이 과정에서 디자인 결정을 하나씩 내려갔어요.

- **폰트**: Geist Sans. 숫자가 많은 대시보드라서 가독성 좋은 산세리프체를 골랐어요. 테이블의 숫자는 JetBrains Mono로 따로 지정했고요
- **색상**: Warm Zinc 톤 기반. 배경은 #f8f8f6 (살짝 따뜻한 회색), 카드는 흰색, 보더는 #e8e5e0. 수익률은 초록(#059669), 손실은 빨강(#dc2626), 강조는 파랑(#2563eb)
- **카드**: 16px 둥글기, 1px 보더, 살짝 그림자. 호버하면 그림자가 좀 더 진해지는 정도
- **간격**: 사이드바 220px, 탑바 44px, 콘텐츠 영역 좌우 32px 패딩

이렇게 하나의 기준을 잡아놓으니까, 나머지 페이지 목업은 이 기준 안에서 변형만 하면 됐어요. 결국 25개의 HTML 목업을 만들었어요.

## 3단계: 디자인 토큰 문서화 — 이게 게임 체인저였어요

목업을 만들고 나서, 이걸 AI에게 어떻게 전달할지 고민했어요. 매번 "이 목업 보고 따라해줘"만으로는 부족하더라고요. AI가 새 페이지를 만들 때 이전 페이지의 스타일을 기억하지 못하는 경우가 있었거든요.

그래서 디자인 토큰을 문서(DESIGN.md)로 정리했어요.

```
색상:
  배경: #f8f8f6
  카드: #ffffff
  보더: #e8e5e0
  텍스트 기본: #1a1a19
  텍스트 보조: #6b7280
  양수: #059669
  음수: #dc2626
  강조: #2563eb

타이포그래피:
  기본 폰트: Geist Sans
  숫자/코드: JetBrains Mono
  KPI 값: 26px, 700
  섹션 제목: 16px, 700
  뱃지: 11px, 600, uppercase

간격:
  카드 내부 패딩: 24px
  카드 간 간격: 20px
  카드 둥글기: 16px
```

이런 식으로 모든 규칙을 명시적으로 적어둔 거예요. AI에게 "이 문서를 참고해서 만들어줘"라고 하면, 어떤 페이지를 만들든 같은 스타일이 나오더라고요.

여기에 더해서, AI가 자주 실수하는 부분을 정리한 가이드(gotchas)도 만들었어요. 예를 들면:

- "HTML을 넣을 때 반드시 `sanitize=False` 옵션을 써야 한다" (안 쓰면 태그가 그대로 보여요)
- "다크 모드 CSS 선택자는 `body--dark`이지 `body.dark`가 아니다" (이거 때문에 다크 모드가 안 먹는 경우가 있었어요)
- "Plotly에서 `add_vline` 쓰면 특정 조건에서 크래시 나니까 `shapes`를 써라"

이런 함정들을 미리 문서화해두니까, 같은 실수를 반복하지 않게 됐어요.

---

## 4단계: 실제 전환 — 생각보다 빠르게

준비가 되고 나니까 실제 전환은 꽤 빨랐어요.

**진행 순서:**

1. **레이아웃 (layout_v2.py)** — 사이드바, 탑바, 다크 모드, CSS 변수. 모든 페이지가 공유하는 껍데기를 먼저 만들었어요
2. **Overview** — 가장 간단한 페이지부터 시작. KPI 카드 5개 + 차트 1개
3. **Backtest Result List** — AG Grid 테이블이 들어가는 복잡한 페이지. 여기서 테이블 스타일 기준을 잡았어요
4. **Backtest Detail** — 가장 무거운 페이지 (차트 6개, 테이블 2개). 이게 되면 나머지는 다 되는 거예요
5. **나머지 페이지들** — 비교, 상관관계, 리스크, 페이퍼 트레이딩, 데이터 소스 8개…

기반이 되는 레이아웃과 디자인 토큰이 잡혀 있으니까, 후반부 페이지들은 앞에서 만든 패턴을 재활용하면 됐어요. "Overview의 KPI 카드 패턴으로 이 페이지도 만들어줘" 이런 식으로요.

## 삽질한 것들

순탄하지만은 않았어요. 기억나는 것들:

**차트 라이브러리 호환 문제**

원래 Chart.js를 쓰고 있었는데, NiceGUI에서는 Plotly가 더 잘 맞더라고요. 근데 Chart.js로 만든 차트를 Plotly로 바꾸면 옵션 이름이 완전히 달라져요. 이것도 변환 가이드를 문서로 만들어서 AI에게 참고하게 했어요.

**다크 모드에서 색이 깨지는 문제**

라이트 모드에서 잘 되던 게 다크 모드로 바꾸면 글자가 안 보이거나, 카드 배경이 이상해지거나. CSS 변수를 쓰는 건 알겠는데, 변수를 빠뜨린 곳이 있으면 그 부분만 색이 이상해지거든요. 결국 CSS 변수를 한 곳에서 관리하고, 모든 페이지가 그걸 참조하도록 구조를 잡아야 했어요.

**"거의 다 됐는데 이 부분만…"의 반복**

AI가 90%를 만들어주면, 나머지 10%를 맞추는 데 생각보다 시간이 걸렸어요. 미세한 여백, 테이블 헤더 정렬, 반응형에서 카드가 깨지는 것 같은. 이런 디테일은 직접 확인하고 피드백을 줘야 했죠.

---

## 돌아보면

**AI는 "반복 작업"에 강해요**

20개 페이지를 같은 디자인 시스템으로 전환하는 건, 본질적으로 반복 작업이에요. 패턴이 정해져 있고, 규칙이 명확하면, AI가 빠르고 정확하게 해줘요. 반면 패턴을 정하는 것 — 어떤 색을 쓸지, 어떤 레이아웃이 이 데이터에 맞는지 — 이건 사람이 해야 하더라고요.

**"AI에게 잘 시키는 법"은 결국 "기획을 잘 하는 법"이었어요**

이번 작업에서 가장 많은 시간을 쓴 건 코드를 짜는 게 아니라, 목업을 만들고 디자인 규칙을 정리하는 거였어요. 그러니까 결국 AI를 잘 활용하려면 **"뭘 만들어야 하는지"를 명확하게 정의하는 능력**이 필요한 거예요. 이건 기획의 본질이잖아요.

프롬프트 엔지니어링이라고 하면 뭔가 기술적인 것 같지만, 실제로 해보면 **"요구사항을 명확하게 전달하는 능력"**이에요. 모호하게 시키면 모호한 결과가 나오고, 구체적으로 시키면 구체적인 결과가 나와요. 이건 개발자한테 기획서를 전달할 때도 똑같은 원리죠.

**완벽하지 않아도, 돌아가면 괜찮아요**

솔직히 말하면, 제가 만든 게 전문 디자이너가 만든 것만큼 예쁘진 않을 거예요. 근데 사용자(리서처)들 반응은 좋았어요. 예쁘냐 아니냐보다 "일관되고 쓰기 편한가"가 더 중요했던 거죠. 완벽을 기다리면 아무것도 안 되고, 80점짜리라도 나오면 그게 0점보다 훨씬 나은 거예요.

## 정리하면

이 세 편의 글에서 저는 이런 이야기를 하고 싶었어요.

1. **리소스가 부족할 때, 직접 할 수 있는 방법을 찾으면 된다** (글 1)
2. **사용자의 말을 그대로 듣지 말고, 진짜 필요한 것을 찾아야 한다** (글 2)
3. **AI는 강력한 도구지만, 결국 기획과 판단은 사람의 몫이다** (글 3)

저는 개발자가 아니에요. 데이터 인프라를 다루고, 도메인을 이해하고, 사용자한테 필요한 게 뭔지 고민하는 서비스 기획자예요. 근데 AI 덕분에 그 고민을 실제 서비스로 만들어낼 수 있었어요.

AI 시대에 기획자의 역할은 없어지는 게 아니라, 오히려 더 중요해진다고 느꼈어요. AI가 코드를 대신 짜주는 시대일수록, **"뭘 만들어야 하는지"를 정의하는 사람**이 더 필요해지니까요.
