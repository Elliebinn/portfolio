## 우리 팀 상황

알파브릿지에서 퀀트 전략을 연구하는 팀은 총 5명이에요. 리서처 4명, 백엔드 개발자 1명. 규모가 작다 보니 각자 맡은 일이 꽤 뚜렷한 편이죠.

저는 그중에서 데이터 인프라와 서비스 기획을 담당하고 있었어요. ClickHouse에 시계열 데이터를 적재하고, MySQL 스키마를 관리하는 데이터 쪽 업무와 함께, 분석 도구의 UX를 개선하는 기획 역할도 맡고 있었죠.

우리 팀에는 매일 쓰는 내부 도구가 하나 있는데, 전략을 백테스트하고 성과를 분석하는 웹 대시보드예요. 수백 개의 전략을 조회하고, 비교하고, 보유 종목 변화를 추적하는 — 리서처들한테는 없으면 안 되는 도구거든요.

문제는 이 도구를 **개선할 사람이 없었다**는 거예요.

프론트엔드 개발자도 없고, 디자이너도 없고, 기획자도 없어요. 리서처들은 분석에 집중해야 하고, 유일한 개발자분은 백엔드 인프라로 바쁘시고. 화면 쪽에 불편한 점이 있어도 계속 "나중에"가 반복되는 상황이었죠.

## 옆에서 보다가, 직접 하기로 했어요

저는 사실 알파브릿지 전에 모회사인 에셋플러스자산운용에서 AI 공모펀드를 직접 운용한 경험이 있어요. S&P500 기반 공모펀드를 출시하고, AI 운용지원 시스템을 기획해서 매매·운용 시간을 83% 단축시킨 적도 있고요. 그래서 리서처들 옆에서 일하다 보면, 단순히 "불편하다"는 말 뒤에 어떤 니즈가 있는지가 보이더라고요.

예를 들면 이런 이야기들이 계속 들렸어요.

> "전략이 200개가 넘는데, 내가 지난주에 돌린 거 찾으려면 한참 스크롤해야 해"
>
> "Holdings 화면에서 날짜를 드래그로 선택하게 되어있는데, 리밸런싱 날짜로 바로 못 가서 불편해"

Holdings 날짜 얘기를 들었을 때, 저는 바로 뭐가 필요한지 알 수 있었어요. 실무에서는 리밸런싱 시점마다 포지션이 어떻게 바뀌었는지를 추적하는 게 핵심이거든요. 날짜 선택 UI를 예쁘게 바꾸는 게 답이 아니라, **리밸런싱 포인트 사이를 바로 이동할 수 있는 기능**이 필요한 거였죠.

이런 건 요청으로 올라오지 않아요. 현업을 해본 사람이어야 보이는 것들이에요.

그래서 생각했어요 — 요청 넣고 기다리는 것보다, 직접 만들어보는 게 빠르겠다.

## 이 도구가 하는 일

간단히 소개하면, 우리 분석 도구는 이런 걸 해요.

**전략 관리 & 분석**
- 수백 개 백테스트 전략을 한 곳에서 조회하고 검색
- 전략별 수익률, 샤프 비율, MDD 같은 핵심 지표 대시보드
- 전략끼리 같은 조건으로 나란히 비교

**포트폴리오 분석**
- 보유 종목 변화를 리밸런싱 포인트별로 추적
- 종목 비중 변화 시각화
- 상관관계, 리스크-리턴 프로파일

**데이터**
- 한국/미국 시장 커버 (KOSPI, KOSDAQ, S&P 500, ETF)
- 수천 종목 × 수년치 일별 데이터
- 시계열 조회용 ClickHouse + 메타데이터용 MySQL, 용도에 맞게 DB를 나눠서 운영

페이지만 20개가 넘고, 한국어/영어 다국어 지원에 다크 모드까지 있어요.

## 근데, 프론트를 어떻게?

솔직히 저는 프론트엔드를 잘 몰라요. 데이터 인프라랑 서비스 기획이 주 영역이니까요.

그런데 요즘은 AI가 있잖아요.

Claude Code라는 AI 코딩 도구를 쓰면, 제가 "이렇게 만들어줘"라고 설명하면 실제 코드를 생성해줘요. 물론 그냥 시키기만 하면 되는 건 아니었어요. 꽤 시행착오가 있었죠.

**처음에는 말로 설명했어요.** "카드 형태로 KPI를 보여주고, 아래에 차트를 넣어줘" 같은 식으로요. 결과가 나오긴 하는데… 제가 원하는 느낌이 아닌 거예요. AI가 "적당히" 만들어주는 화면은 어딘가 어설프더라고요.

**그래서 HTML 목업을 직접 그렸어요.** 디자이너가 없으니까 제가 직접 색상, 폰트, 간격, 카드 레이아웃을 정하고, HTML로 화면을 먼저 만들었어요. 이걸 AI에게 주면서 "이 목업이랑 똑같이 만들어줘"라고 하니까, 결과물이 확 달라졌죠.

여기서 하나 더 깨달은 게 있어요. 디자인 기준을 **문서로 정리해두지 않으면** AI가 페이지마다 다른 스타일을 만들어요. 그래서 색상 코드, 폰트 크기, 간격, 카드 스타일 같은 디자인 토큰을 한 곳에 정리해뒀어요. 이렇게 하니까 20개 넘는 페이지가 일관된 느낌으로 나오더라고요.

정리하면 이런 흐름이었어요.

1. 리서처들 불편사항 수집 + 도메인 경험으로 진짜 니즈 파악
2. HTML 목업으로 화면 설계
3. 디자인 토큰 정리 → AI에게 전달
4. AI가 생성한 코드 확인 → 피드백 → 수정 반복
5. 실제 데이터로 테스트 → 리서처 피드백 반영

## 결과적으로

- 20개 이상의 분석 페이지를 새로운 디자인으로 전환했어요
- 태그 기반 전략 분류, 리밸런싱 포인트 네비게이션 같은 실무 기능을 추가했고요
- 일관된 디자인 시스템도 만들었어요 (다크 모드, 다국어 포함)
- 계속 "나중에"로 미뤄지던 개선들이 실제로 반영됐죠

## 돌아보면서 느낀 것들

**도메인을 아는 게 이렇게 중요하구나**

사용자가 "불편해요"라고 말할 때, 그 뒤에 진짜 뭐가 필요한지를 파악하려면 그 업무를 이해해야 하더라고요. 저는 에셋플러스에서 공모펀드를 직접 운용해봤기 때문에, 요청된 것이 아니라 필요한 것을 설계할 수 있었어요. 같은 요청이라도 도메인을 아는 사람이 만들면 결과가 달라지는 걸 느꼈죠.

**AI를 잘 쓰려면, 먼저 뭘 만들지 알아야 해요**

Claude Code로 코드를 생성하는 건 빨라요. 그런데 "이 화면에서 사용자가 뭘 해야 하는지", "이 데이터를 어떤 차트로 보여줘야 하는지"는 AI가 판단하지 못해요. 기획과 판단이 먼저이고, AI는 그걸 빠르게 실현해주는 도구라는 걸 계속 느꼈어요. 잘 쓰는 것만큼 **어디에서 안 쓸지** 아는 것도 중요하더라고요.

**기술을 이해하면 소통 비용이 줄어요**

기능 하나를 요청하려면 보통 "이런 거 되나요?" → "가능한데 오래 걸려요" → "그러면 우선순위가…" 이런 사이클을 거치잖아요. 기술을 이해하면 이 과정이 확 줄어들어요. 가능한 것과 어려운 것을 구분할 수 있고, 대안을 직접 제시할 수도 있으니까요.
